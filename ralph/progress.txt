## Codebase Patterns
- SQLAlchemy 2.0 declarative base lives at `server/src/database.py` (`Base`, `engine`, `SessionLocal`, `get_database_url`). All models import `Base` from there.
- ORM models live at `server/src/models/<name>.py`; `server/src/models/__init__.py` is the single import point and re-exports `Base` plus every model so Alembic's `env.py` can register metadata via one import.
- Alembic is configured at `server/alembic.ini` with `prepend_sys_path = %(here)s/..` so migrations always run from repo root: `alembic -c server/alembic.ini upgrade head`.
- `env.py` pulls the DB URL from `DATABASE_URL` env var (default `sqlite:///./server_dev.db`) and sets `render_as_batch=True` for SQLite so schema changes remain portable.
- Absolute imports (`from server.src.<pkg>`) are mandatory — requires `server/__init__.py` + `server/src/__init__.py` present. Relative `from src...` breaks mypy's dual-namespace check.
- Enum-like fields use VARCHAR + `CheckConstraint("col IN (...)")` for Postgres/SQLite portability, and values are re-exported as a module-level tuple (e.g. `SESSAO_STATUS_VALUES`) for reuse in API layers.
- Mypy config at repo root `mypy.ini` covers `server/src` + `server/alembic` with strict-ish defaults (`disallow_untyped_defs = True`). Run `mypy` from repo root.
- FastAPI app object is at `server/src/main.py:app`; boot with `uvicorn server.src.main:app --reload` from repo root.
- Pydantic v2 schemas live at `server/src/schemas/<entity>.py`. Use `EmailStr` (requires `email-validator`) for emails and `ConfigDict(from_attributes=True)` on response models so SQLAlchemy ORM rows convert via `Model.model_validate(orm_obj)`.
- HTTP routers live at `server/src/routers/<name>.py` and are wired into `server/src/main.py` via `app.include_router(<module>.router)`. Use `APIRouter(prefix="/api/<resource>", tags=[...])`.
- DB-bound endpoints inject the session with `SessionDep = Annotated[Session, Depends(get_session)]` — keeps mypy clean (no `Depends()` default-value type mismatch).
- Password hashing helpers at `server/src/security.py` — `hash_password(senha)` / `verify_password(senha, senha_hash)` use bcrypt's `gensalt()`/`checkpw`. Always hash before persisting; never store plaintext.
- Smoke-test FastAPI routers without spawning uvicorn: build a `TestClient(app)` over an in-memory SQLite engine and override `get_session` via `app.dependency_overrides`. In-memory SQLite **must** use `poolclass=StaticPool` + `connect_args={"check_same_thread": False}`, otherwise each connection sees an empty schema.
- JWT auth: `server/src/security.py` has `create_access_token(subject)` / `decode_access_token(token)` using `HS256` + `SECRET_KEY` env var (dev default present). `server/src/auth.py` exposes `get_current_user` dependency and the `CurrentUser = Annotated[Usuario, Depends(get_current_user)]` alias — protect routes with `def route(user: CurrentUser, ...)`. The dependency uses `OAuth2PasswordBearer(auto_error=False)` so all 401 responses share a single uniform body (no leak of "missing token" vs "bad token").
- Heavy ML deps (`cv2`, `ultralytics`) are isolated in `server/src/video_pipeline.py` behind the `VideoValidator` protocol; `get_video_validator()` is the FastAPI dependency, overridable in tests. Lazy-import `cv2`/`ultralytics` inside methods so importing the module never loads YOLO. `mypy.ini` has `ignore_missing_imports = True` for `cv2.*`, `ultralytics.*`, `numpy.*`.
- Multipart form uploads need `python-multipart` (in `requirements.txt`). Use `Annotated[float, Form()]` / `Annotated[UploadFile, File()]` on endpoint params; persist to `UPLOAD_DIR` (env var, default `./uploads/`, gitignored) and call `_cleanup(path)` on any validation failure before re-raising `HTTPException`.
- Dispatch long-running work with `background_tasks: BackgroundTasks` parameter (FastAPI built-in) + `background_tasks.add_task(fn, ...)`; tasks run after the response is sent so the client is not blocked. The pipeline entrypoint convention is `server.src.video_pipeline.run_pipeline(sessao_id, video_path)`.
- Enforce per-user access on resource routes by loading the row (`session.get(Model, id)`) then: `None → 404`, `row.usuario_id != user.id → 403`. Do not leak existence by collapsing to 404 for both — the tests (and PRD) require a distinct 403.
- Background pipeline runs **outside** FastAPI's dependency-injection scope, so `app.dependency_overrides` does not affect it. `server.src.video_pipeline.run_pipeline` takes optional `extractor` (`PoseExtractor` protocol) and `session_factory` kwargs — injection points for tests. The HTTP call site keeps the positional signature `run_pipeline(sessao_id, video_path)`.
- Heavy-pipeline protocols follow the same lazy pattern as `VideoValidator`: `cv2`/`ultralytics` imports live inside method bodies, never at module top, so `import server.src.video_pipeline` never loads YOLO.
- COCO keypoint filtering: `KEYPOINT_SCORE_THRESHOLD = 0.5` — below that, set keypoint to `None` (treated as NaN downstream). A frame is "low quality" when ≥ `LOW_QUALITY_KP_THRESHOLD` (= 9, majority of 17) of its keypoints are None. Reuse these constants from `video_pipeline` for any downstream biomechanics code.
- `smooth_frames(frames, window=5)` applies a moving-average on x,y per keypoint, **preserving None** — it never interpolates over a dropped frame. Any new keypoint-consuming story should accept the smoothed series (post-suavização) as input, not raw extractor output.
- Biomechanics utilities live at `server/src/biomechanics/<metric>.py`; each module re-exports its public API via `server/src/biomechanics/__init__.py`. The package is the single import point for downstream stories (US-008 onward) and for `run_pipeline` when it wires up metric persistence.
- Unit tests for biomechanics go in `server/tests/test_<metric>.py` (pytest). Run with `python -m pytest server/tests/` from repo root. `server/tests` is included in `mypy.ini` so test helpers must be fully typed (same strictness as `server/src`).
- Domain errors from biomechanics utilities raise `ValueError` with a module-level `MSG_*` constant as the message, so callers can match on identity and tests use `pytest.raises(ValueError, match=re.escape(MSG_*))`.
- Métricas calculadas por `run_pipeline` são gravadas em `METRICA` via funções privadas `_persistir_<metric>(session, sessao_id, frames)` (lazy-import do módulo `biomechanics` + `Metrica` dentro da função para manter `video_pipeline` leve). Não alteram `sessao.status` — a transição final para `concluido` pertence a US-016 (gerador de recomendações).
- Convenção de ângulo do joelho no **backend** (US-008+) é o ângulo interno (perna estendida = 180°), diferente da convenção `180-internal` do legado em `processing/src/mainGraph.py`. Ao tocar qualquer módulo biomecânico novo, mantenha a convenção interna (arccos do produto escalar normalizado).
- Detecção de contato inicial: picos locais estritos de Y do tornozelo (helper privado em `biomechanics/joelho.py`). Bordas nunca são pico. Reutilize esta ideia para US-010 (cadência) e US-013 (TCS).
- Métricas por-lado **sem ciclos** (ângulo médio de cotovelo, etc.) usam o shape `AnguloXxxLado(angulo_medio_graus, frames_validos)` — campo `frames_validos: int` em vez de `frames_contato: tuple[int, ...]`. Reserve `frames_contato` para métricas de ciclo (joelho no contato, TCS, overstriding).
- Cálculos dependentes de tempo real (cadência, TCS) recebem `fps` como parâmetro explícito de `calcular_<metric>(frames, fps)` e levantam `ValueError(MSG_FPS_INVALIDO)` para `fps <= 0`. Os helpers `_persistir_<metric>` em `video_pipeline.py` repassam `result.fps` — a camada biomecânica nunca lê `sessao.fps` nem toca o ORM.
- Convenção de "flexão" em filtros do PRD: `flexao_graus = 180 - angulo_interno`. As métricas que filtram por fase de apoio (tronco em US-011, futuras métricas de TCS) traduzem internamente as faixas do PRD ("flexão ~40°–45°") para o ângulo interno backend. Constantes `FLEXAO_APOIO_MEDIO_{MIN,MAX}_GRAUS` em `biomechanics/tronco.py` são reutilizáveis.
- Métricas com sinal (inclinação do tronco) preservam direção via convenção: positivo ⇒ +X (frente para corredor andando para a direita), negativo ⇒ -X. A fórmula `arctan(ΔX/ΔY)` do PRD é aplicada como `arctan(ΔX / -ΔY_imagem)` para compensar o eixo Y crescente para baixo nas coordenadas de imagem; ΔY_imagem ≥ 0 (tronco horizontal/invertido) resulta em frame descartado.
- Métricas que precisam de `fator_escala` (overstriding em US-012, oscilação vertical em US-014) recebem o `fator_escala: float` como parâmetro puro; é o `_persistir_<metric>(session, sessao_id, frames, altura_cm)` que carrega `usuario.altura_cm` (de `sessao.usuario`), chama `calcular_fator_escala(frames, altura_cm).fator_escala` e repassa. A camada biomecânica nunca lê o ORM nem `usuario.altura_cm` direto; falhas do fator (altura None/≤0 ou nenhum frame válido para altura_pixels) são engolidas no helper persist (no-op silencioso) — não interrompem o pipeline.
- `_find_initial_contact_indices` + `_ankle_y` agora estão duplicados em três módulos (`joelho.py`, `cadencia.py`, `overstriding.py`). US-013 (TCS) será a quarta cópia — neste ponto vale extrair para `biomechanics/_picos.py` e reexportar dos módulos atuais. Adiar até US-013 para manter cada história isolada (refatorações cross-module devem ter sua própria iteração).
- TCS (US-013) usa detecção de ciclo diferente das demais métricas de contato: **plateau é aceito** (`Y[i-1] < Y[i]` e `Y[i+1] ≤ Y[i]`) e o fim do ciclo é a primeira queda estrita (`Y[j] < Y[j-1]`). O helper privado `_detectar_ciclos_contato(y_series)` retorna `(inicio_idx, frames_de_contato)`. Isso é incompatível com `_find_initial_contact_indices` (picos estritos sem plateau) — uma refatoração cross-module futura precisa expor os dois modos (pico estrito vs. plateau) em vez de substituir o atual por um só.
- Métricas informativas (`apenas_informativa=True`) já acumulam duas histórias: cadência (US-010) e TCS (US-013). US-016 (nota geral) deve filtrar em bloco todas as métricas com esse flag para evitar penalização — não o fazer case-a-case por `tipo`.
- Métricas de **ciclo de passada** (oscilação vertical em US-014) usam contatos consecutivos do **pé direito** como bordas de ciclo (mesma convenção da cadência por PRD Seção 6.3). Por ciclo, o intervalo de frames é fechado `[i, j]` (inclui ambos os contatos). Quando a métrica precisa de N contatos para formar N-1 ciclos, o guard correto é `len(contatos) < 2 → None`.
- Índice de simetria (`indice_simetria(esq, dir)`) em `biomechanics/simetria.py` é a única fórmula reutilizável esq/dir do PRD: `|esq-dir| / ((esq+dir)/2) × 100`. Retorna `None` para lado ausente (erro controlado) e `0.0` quando `esq == dir == 0` (degenerado). `calcular_simetria(frames, fps, fator_escala)` degrada graciosamente: `fps <= 0 ⇒ simetria_tcs=None`; `fator_escala is None or <= 0 ⇒ simetria_oscilacao=None`. Reutilizável para qualquer métrica futura esq/dir (overstriding, ângulos de cotovelo) se US-016/US-025 pedir.
- Gerador de recomendações (`biomechanics/recomendacoes.py`) é agnóstico a keypoints: recebe apenas `list[tuple[str, float]]` (tipo, valor) e devolve `AnaliseResultado(nota_geral, feedback_ia, recomendacoes)`. O dict de dispatch `_ANALISADORES` mapeia cada `tipo` (coluna `Metrica.tipo`) à função de análise — novas métricas, quando adicionadas, só precisam registrar a entrada no dict + função `_classificar_*`/`_analisar_*`. Tipos desconhecidos são silenciosamente ignorados (NO-OP). Pesos: `PESO_ATENCAO=0.5`, `PESO_CRITICO=1.5`, `NOTA_MAXIMA=10.0` — ajustar aqui reflete em toda a nota.
- `_finalizar_analise(session, sessao_id)` em `video_pipeline.py` é o **único** ponto que transita `sessao.status` para `concluido`. Demais `_persistir_<metric>` jamais tocam o status. Ele carrega todas as `Metrica` da sessão, filtra `apenas_informativa=True` em Python (sem SQL `where == False` para manter portabilidade), e grava `Recomendacao` + atualiza `nota_geral` + `feedback_ia` + `status`. Endpoints que leem esses campos (US-017, US-019, US-025) devem tratá-los como populados apenas quando `status='concluido'`.
- Endpoints de análise (sob prefixo `/api/analysis`) vivem em `server/src/routers/analysis.py`. Controle de acesso segue o mesmo padrão de vídeos: `session.get(SessaoAnalise, id)` → 404 se `None`, 403 se `usuario_id != user.id`. Para status de erro (`erro_qualidade_keypoints`, `erro_multiplas_pessoas`), responder 200 com um campo `erro` descritivo (usar `STATUS_DESCRICAO` de `routers/videos.py` como fonte única da mensagem) e listas vazias para `recomendacoes`/`metricas_resumidas`. Schemas de resposta (`RecomendacaoResponse`, `MetricaResumida`) usam `ConfigDict(from_attributes=True)` para conversão direta de ORM rows via `Model.model_validate(orm_obj)`.
- Dados brutos por-frame (para `/api/analysis/{id}/raw`) são serializados como JSON em `SessaoAnalise.dados_brutos_json` (TEXT, nullable) pelo helper `_persistir_dados_brutos` em `video_pipeline.py`. A serialização acontece **antes** dos demais `_persistir_<metric>`, logo após o status transitar para `calculando_metricas`; não é gravada em caminhos de erro (`erro_qualidade_keypoints`/`erro_multiplas_pessoas`). Valores ausentes (keypoints com score < 0.5, métricas per-frame sem dados) viram `null` na serialização — nunca `NaN` (uso de `json.dumps(..., allow_nan=False)`).
- Métricas **per-frame** (ângulo joelho/cotovelo/tronco + y_com) vivem em `server/src/biomechanics/frame_metrics.py` separadas das agregadas (média, por-ciclo, etc.). Convenção: cada função recebe um único `FrameKeypoints` + lado (quando aplicável) e retorna `float | None`. Per-frame não aplica filtros contextuais (ex: `inclinacao_tronco_frame` não filtra por apoio médio — vale para qualquer frame com os quatro keypoints disponíveis). Reaproveite estes helpers para US-026 (gráficos por frame) e para qualquer história futura que precise de séries temporais sem agregação.
- Tests de roteadores FastAPI seguem o padrão `server/tests/test_<router>_<endpoint>.py` com fixtures pytest: `session_factory` (in-memory SQLite + StaticPool) e `client` (TestClient com `app.dependency_overrides[get_session]`).
